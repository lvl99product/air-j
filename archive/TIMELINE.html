Phase 01 Foundation & Core Value Prop
Early 2026 Product Strategy
Strategic Bet: The Zero-Friction Utility
How do we build a task manager that is instantly accessible to anyone, anywhere, with no setup barriers? We made a strategic decision to build air-J as a pure client-side application using vanilla HTML/CSS/JS and localStorage. By eliminating server infrastructure, databases, and build pipelines, we ensured the application is instantly deployable, lightning-fast, and works completely offline.
Product Strategy Zero-Friction UX Browser-Native Local Persistence
Strategic Trade-off: Choosing vanilla JS over a modern framework like React or Vue. While frameworks speed up component rendering, they introduce build steps and dependencies. A framework-free approach guarantees zero-friction hosting on GitHub Pages and runs immediately on any browser.
Early 2026 Design System
Premium Branding & Design Language
To stand out in a crowded market of generic productivity tools, we established a premium visual identity. We designed a modern dark-mode system using high-contrast typography (Outfit + Plus Jakarta Sans) and a vibrant indigo-to-teal gradient palette. The focus was on building key structural blocks—the multi-column Kanban board, category tags, and task detail views—as reusable, scalable design components.
Visual Identity Dark Mode Typography Design System
Early 2026 Value Proposition
The Differentiator: Dynamic Dual-Mode UX
Our user research showed that teams are often forced to switch tools as they scale, moving from simple, free-flow Kanban to structured agile Scrum. We designed a dual-mode architecture that lets users switch between Kanban and Scrum workflows seamlessly. The product challenge was maintaining a single unified task database while dynamically adjusting UI complexity—introducing backlog grooming, sprint indicators, and burndown controls only when in Scrum mode.
Dual-Mode Architecture Workflow Flexibility Market Differentiator Scrum/Kanban Toggle
Strategic Outcome: The dual-mode architecture became air-J's core differentiator, providing a single source of truth that dynamically adapts to the user's process complexity without cluttering the interface.

Phase 02 Mobile UX & Layout Exploration
May 2026 UX Exploration
Squeezing a Kanban Board into a Mobile Screen
A horizontal, multi-column Kanban board is notoriously difficult to use on mobile viewports. To solve this layout constraint, we drafted and built four interactive prototypes. This structured UX exploration allowed us to evaluate the usability of horizontal swipe gestures, accordions, bottom sheets, and dashboard summaries in real-time.
Option 1: Horizontal Swipe
Swipe horizontally between columns. Natural gesture but hides board context.
Option 2: Accordion Stack
Collapsible vertical stacks. Simple hierarchy, but hides overall board status.
Option 3: Drawer Sheets
Focused primary view with bottom sheets for secondary columns. High layout complexity.
Option 4: Dashboard Focus ✓ Selected for Test
A high-level status overview with contextual links to columns. Selected to stress-test Scrum workflows.

May 2026 Usability Test
Stress-Testing the Dashboard Layout
We selected the Dashboard-First layout (Option 4) for a second round of deeper prototyping. Our goal was to stress-test how a user would perform complex sprint-planning tasks on a small screen. By creating an interactive high-fidelity prototype, we discovered that while the dashboard worked well for viewing progress, it added too much navigation overhead for active backlog grooming and task staging.

Phase 03 Sprint Planning UX & Mechanics
June 2026 Interaction Model
Designing the Planning Experience
Sprint planning is a multi-step, content-heavy process: reviewing backlog items, sizing them, and committing them to a sprint. On mobile, this usually results in high cognitive load and constant screen toggling. We prototyped and compared three distinct interaction models inside device mockups to find a fluid, low-friction planning experience.
Option 1: Native Tabs
Separate tabs for Backlog and Active Sprint. Simple but causes high context-switching friction.
Option 2: Sprint Cart ✓ Shipped
A persistent floating "Cart" banner. Users browse the backlog and "stage" tasks from anywhere.
Option 3: Guided Wizard
A step-by-step full-screen flow. Great for onboarding, but too rigid for power users.
Strategic Design Rationale: We chose Option 2 (Sprint Cart) because it applies the familiar e-commerce "shopping cart" mental model to agile planning. Users can groom their backlog and stage items in a persistent bottom sheet, keeping their visual context intact. Sliding up the cart allows them to review, prioritize, and initiate the sprint in one fluid gesture.

June 2026 Implementation
Iterating from Prototype to Production
We integrated the Sprint Cart into the production codebase, refining the user experience through collaborative cycles of user feedback:
  • Layout Density Optimization: Addressed visual clutter concerns by removing redundant sprint headers (e.g. removing "Sprint 9" labels) and condensing the active sprint widget to a clean "Active" badge.
  • Frictionless Backlog Entry: Users asked how to add unclassified items quickly. We built a direct "Quick Add" input inside the backlog panel, deferring categorization to later grooming phases.
  • Actionable Empty States: Designed interactive empty states that explicitly guide users to add tasks or stage items, keeping the cart helper visible even before the sprint begins.
Collaborative Feedback Information Density Frictionless Quick-Add Actionable Empty States
Product Outcome: The Sprint Cart successfully shipped as the core planning mechanism. It balances visual simplicity with functional depth, enabling complete sprint planning directly from a mobile screen.

Phase 04 Quality Assurance & Release
June 2026 Quality Assurance
Polishing Mobile Interactions & Edge Cases
We conducted a rigorous polish cycle to squash visual bugs and interaction conflicts on smaller viewports:
  • Component Scoping: Scoped the "Next Sprint" sticky button strictly to the Scrum active view to prevent it from cluttering the Kanban board.
  • Overlap Mitigation: Adjusted the floating action button (FAB) positioning to eliminate overlap with the persistent Sprint Cart banner.
  • Screen Real Estate: Optimized vertical padding and spacing on mobile to prevent excessive scrolling and maximize visible task cards.

June 2026 Release
Compliance, Security, and Production Launch
Prior to publishing the code to GitHub, we conducted a comprehensive compliance and security audit. We sanitized all personally identifiable information (PII), absolute local file paths, and environment placeholders. We configured a strict .gitignore and defined automated pre-push security verification guidelines to protect code integrity.
Compliance Audit Security Guards PII Sanitization Release Readiness
Production Launch: The repository was launched clean at github.com/lvl99product/air-j. Combined with GitHub Pages, we achieved our initial product goal: a fully functional, offline-capable task manager served instantly with zero deployment overhead.
