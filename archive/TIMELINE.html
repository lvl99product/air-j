Phase 01 Foundation & Core Architecture
Early 2026 Milestone
Project Inception
air-J started as a single-page Kanban board — a pure client-side application with no backend or database. The core decision was to keep the entire product as a browser-native experience using localStorage for persistence, eliminating all infrastructure complexity.
HTML CSS Vanilla JS localStorage
Design Decision: No frameworks. Pure HTML/CSS/JS gives zero build-step complexity, making it instantly hostable on GitHub Pages and accessible to anyone without a Node.js environment.
Early 2026 Feature
Kanban Board & Branding
Established the core Kanban columns (Backlog → Todo → Doing → Done), the dark-mode design system with Outfit + Plus Jakarta Sans typography, and the signature indigo/teal gradient brand palette. The sidebar navigation, task creation modal, and category management system were all built in this phase.
Kanban Design System Dark Mode Categories Task Modal
Early 2026 Architecture
Scrum Mode Added
A mode toggle was introduced to switch between Kanban (free-form flow) and Scrum (sprint-based) workflows. In Scrum mode, a dedicated Backlog column appears alongside the active sprint columns, and a sprint management system tracks start/end dates, progress, and completion stats.
Scrum Mode Mode Toggle Sprints Backlog
Outcome: The dual-mode architecture became the defining differentiator of air-J — giving users one tool that handles both lightweight task tracking and structured agile sprints.

Phase 02 Mobile UX Exploration
May 2026 Prototype
Mobile UX Options — Round 1
The first round of mobile exploration tackled the challenge of adapting the multi-column Kanban/Scrum layout to a single-column phone screen. Four layout strategies were explored and presented as interactive prototypes.
Option 1
Horizontal scroll between columns with a segmented tab control at the top.
Option 2
Collapsible accordion sections, each column stacked vertically.
Option 3
Bottom sheet drawers for secondary columns, focused primary view.
Option 4
Dashboard overview mode with contextual column access via floating actions.

Mobile UX Options — Round 2 (Detailed)

Phase 03 Sprint Planning System Design
June 2026 Prototype
Sprint Planning UX — Three Approaches
The core design challenge: how should a user plan a sprint on mobile? Three distinctly different interaction models were prototyped and presented side-by-side in a device mockup showcase for evaluation.
Option 1
Native Tabs
Backlog and Sprint as top-level tab views. Familiar, straightforward navigation.
Option 2 ✓ Chosen
Sprint Cart
A persistent floating "Sprint Cart" banner lets users stage tasks for the next sprint from anywhere in the app.
Option 3
Planning Wizard
A step-by-step guided full-screen wizard walks users through sprint setup sequentially.
Why Option 2 (Sprint Cart)? It allows users to plan the sprint contextually — adding tasks to the cart while browsing the backlog — rather than requiring a dedicated mode switch. The floating banner is persistent but unobtrusive, and the sheet slide-up felt native and fluid on mobile.
View All Three Prototypes
June 2026 Feature
Sprint Cart Implemented in Production
Option 2 was built into the main src/ application. Key refinements during implementation included: the "Active" sprint header widget replacing a bulkier sprint indicator, empty-state guidance for the backlog and sprint planning screen, and a floating cart banner that only appears in Scrum mode. A quick-add task field was also added directly to the backlog for rapid entry without opening the full task modal.
Sprint Cart Active Widget Empty States Quick Add Scrum Mode
Shipped: Sprint Cart lives in production as the primary sprint planning mechanism. Users can stage tasks, review, and lock in the sprint — all without leaving their current view.

Phase 04 Polish, Cleanup & GitHub Launch
June 2026 Fix
Bug Fixes & Responsive Refinements
Multiple rounds of bug fixes addressed mobile-specific issues: the Next Sprint sticky component was scoped to Scrum tab only, the floating action button (FAB) was repositioned to avoid overlapping the Sprint Cart banner, the active sprint widget was condensed to save vertical space, and the sprint cart was made visible even on the "No Sprint Started" screen with contextual guidance copy.
Mobile FAB Sprint Widget Responsive Empty State
June 2026 Milestone
Pre-launch Security Review & Repository Setup
Before the first public push to GitHub, a full PII and security audit was performed across all files. Personal names, absolute local file paths, and developer credentials were removed or anonymised. A .gitignore was created to block macOS system files, Node modules, and environment secrets from ever entering version control. A mandatory pre-push security scan rule was written into the project's agent configuration.
Security Audit .gitignore PII Removal GitHub
Outcome: First public commit pushed to github.com/lvl99product/air-j. GitHub Pages configured to serve the live application from the src/ directory.
